## The idea
Inspired by a TikTok showing friends arguing over who liked a particular video in a shared "Liked Videos" tab ‚Äî reimagined as a small multiplayer party game. Players secretly submit videos they've liked; the group guesses who submitted each one as it's drawn.

## How the game works
1. Players submit links to videos they've personally liked into their own private pool (no list of your own past submissions is ever shown back to you). Other players don't learn who submitted a video until it's drawn into a game and revealed ‚Äî Admin can see the mapping anytime, for moderation.
2. Players join or create a **lobby** via a 6-digit code and land in a waiting room until the Organizer starts a game.
3. A game is a fixed **8 rounds**. Each round draws an unused video from the members' combined pools; everyone guesses who liked it; once everyone's guessed, the answer and updated standings reveal at once.
4. After round 8, final standings show. The same lobby can replay ‚Äî standings reset to zero each new game.

## Why self-submission (not scraping TikTok)
TikTok has no public API for reading a user's Liked Videos or inbox/DMs. Scraping a logged-in session (directly, or via a shared "burner" account) would violate TikTok's ToS, risk a ban, and rely on fragile private endpoints ‚Äî ruled out entirely. Instead, each player pastes a link to their own liked video directly into the app (a normal DB write, zero TikTok contact). Owner and submitter are always the same person.

*Possible future enhancement, not MVP:* a PWA Web Share Target so Android/Chrome users can tap Share on a TikTok video and land pre-filled in the submit form (iOS/Safari doesn't support this ‚Äî manual paste stays as the universal fallback either way).

## Roles
- **Guest** ‚Äî hasn't registered/logged in yet. Can only register or log in; every other tab is visible but locked until then.
- **Player** ‚Äî submits videos, joins/creates lobbies, guesses each round, views standings, edits their own profile, can leave a lobby.
- **Organizer** ‚Äî per-lobby (not global). Starts as the creator; if they leave, the role transfers to whoever joined that lobby earliest among the remaining members. Can start games, invite via the join code, and **kick a player ‚Äî waiting-room only, not mid-game** (not a ban; they can rejoin with the code). Does not manage the video pool ‚Äî that's Admin's job.
- **Admin** ‚Äî global, just the trainee's own account (`is_admin` flag). Can view/close any lobby, manage the player roster (including deleting an account entirely ‚Äî a destructive action, confirmed before it happens), and moderate any submitted video.

Diagram: [`roles-use-cases-diagram.md`](./roles-use-cases-diagram.md) / `.mmd`.

## Lobby lifecycle
Lobbies are temporary ‚Äî they only exist while they have ‚â•1 member.
- Leaving/kicking works the same way: remove that person's membership; if they were Organizer, the role transfers per the rule above.
- **Last member leaves ‚Üí the lobby soft-closes** (`is_open = false`, row kept). Its guesses/points/standings are wiped (always scoped to a live lobby, never persist beyond it) ‚Äî but every submitted video (`posts`) survives regardless of which lobby held it, unless Admin deletes it directly.
- **No joining mid-game:** once Start Game is clicked, that lobby's code stops working for new members until the game finalizes.

## Database schema (Nhost/Postgres via Hasura, 7 tables)
- **`profiles`** ‚úÖ **built 2026-07-08** ‚Äî `id` (PK, ‚Üí `auth.users.id`, cascade delete), `is_admin` (bool, default false). No `display_name` here ‚Äî Nhost's own `auth.users` already has `display_name`/`avatar_url`, so we didn't duplicate it; a Hasura select permission exposes just those two columns of `auth.users` to any logged-in player instead. A Postgres trigger (`on_auth_user_created`) auto-creates the matching `profiles` row on every signup, so app code never has to remember to.
- **`lobbies`** ‚úÖ **built 2026-07-08** ‚Äî `id`, `code` (unique 6-digit), `name`, `organizer_id` (‚Üí profiles, mutable/current), `created_by` (‚Üí profiles, immutable/history), `created_at`, `is_open`.
- **`lobby_members`** ‚úÖ **built 2026-07-08** ‚Äî `lobby_id`, `user_id`, `joined_at` (drives Organizer-transfer order and "last member left" detection).
- **`posts`** ‚úÖ **built 2026-07-08** ‚Äî a submitted video: `submitted_by` (‚Üí profiles, `ON DELETE SET NULL` ‚Äî if an account is later deleted, its past videos stay in history as an orphaned/"removed user" row rather than corrupting old games), video URL, timestamp. No `lobby_id` ‚Äî it's a personal pool. Never deleted automatically ‚Äî only Admin can remove one, via moderation (built 2026-07-08, see #7 below). **`submitted_by` is not exposed to the `user` role at all** ‚Äî the whole secret-submitter mechanic depends on that column never being queryable directly (see reveal mechanism below). `UNIQUE (submitted_by, video_url)` prevents a person submitting the same link twice ‚Äî deliberately scoped per-person, so two different friends independently liking the same video is still allowed (github.com/chrylox/who-liked-it/issues/24).
- **`games`** ‚úÖ **built 2026-07-08** ‚Äî one row per game session in a lobby: `id`, `lobby_id`, `started_at`, `finalized_at` (null while in progress), `round_count`. Revised from a fixed "always 8" ‚Äî **`round_count` now shrinks to fit the pool** (`LEAST(8, distinct videos available across current lobby members)`), so a small/new group can still play instead of being blocked. A partial unique index guarantees only one *active* (unfinalized) game per lobby at a time.
- **`game_rounds`** ‚úÖ **built 2026-07-08** ‚Äî `id`, `game_id`, `round_number`, `post_id` (the video drawn), `revealed_at`.
- **`guesses`** ‚úÖ **built 2026-07-08** ‚Äî `round_id`, `guesser_id`, `guessed_user_id`, `created_at`, `correct` (bool, computed server-side by trigger ‚Äî never client-set). Primary key is `(round_id, guesser_id)`, no surrogate id, matching the original schema note (one guess per person per round).

**How starting/playing a game actually works (all server-side, not client-driven inserts):**
- **Drawing rounds is a Postgres function (`start_game`), not a client INSERT.** A client that could freely INSERT `games`/`game_rounds` rows could rig the draw (e.g. an Organizer always drawing a video whose submitter they already know). `start_game(lobby_id)` is called as a Hasura *mutation*; it uses Hasura's `session_argument` feature to receive the caller's real user id server-side (can't be spoofed the way a plain argument could), checks they're that lobby's Organizer, computes `round_count = LEAST(8, pool size)`, and atomically inserts the game plus all rounds with `ORDER BY random()`.
- **A submitter never guesses on their own round** (confirmed with the trainee 2026-07-08) ‚Äî enforced twice: the reveal-trigger's "expected guesser count" excludes them, and a guess `INSERT` from the actual submitter is rejected outright by a trigger, so the UI *and* the API both prevent it.
- **Reveal is fully automatic, not an explicit action.** A trigger on `guesses` (`AFTER INSERT`) counts actual guesses for that round against the expected count (lobby member count minus the submitter) and sets `revealed_at = now()` the moment they match ‚Äî no client ever sets `revealed_at`. When the round that was just revealed is the game's last round, the same trigger sets `games.finalized_at`, auto-finalizing.
- **Guessing is sequential, enforced by trigger, not just UI:** a guess on round *N* is rejected if any round before it in the same game isn't revealed yet, and rejected outright if round *N* itself is already revealed.
- **Joining is locked for the duration of a game; leaving is not.** `lobby_members` insert (join) still requires *no* unfinalized game for that lobby ‚Äî otherwise the "expected guesser count" the reveal trigger relies on could shift mid-game from someone new showing up. But a player can always voluntarily leave (forfeit), including mid-game ‚Äî a real bug found while building the frontend (#29): the lock originally blocked *any* deletion during a game, but only kicking (an organizer removing someone else) was ever meant to be waiting-room-only. Self-leave works because the reveal trigger recalculates the expected-guesser count fresh on every guess, so a shrinking lobby mid-round is handled gracefully. Everything unlocks again once the game finalizes (verified with a real replay).
- **`posts.submitted_by` is only ever revealed through the `correct` guess field after `revealed_at` is set** (`guesses` select permission requires `round.revealed_at IS NOT NULL`) ‚Äî nobody, including the guesser themselves, can peek at whether their own just-submitted guess was correct before everyone's guessed and the round officially reveals.
- **`game_rounds.is_my_video`** (a Hasura computed field, added while building the frontend) lets the client know "is this round's video mine?" ‚Äî needed so the UI can skip showing you a guess prompt on your own round ‚Äî without that boolean ever revealing *whose* video it is when it isn't yours.
- **Showing other players' names required a fix:** the natural path (`lobby_members`/`guesses` ‚Üí `profiles` ‚Üí `auth.users`) is blocked for anyone but yourself, since `profiles` select is restricted to your own row ‚Äî the intermediate hop just returns nothing for other people. Added direct manual relationships (`lobby_members.member`, `guesses.guesser_account`, `guesses.guessed_account`) straight to `auth.users`, bypassing `profiles` entirely for display purposes.

Standings = aggregate `guesses.correct` grouped by guesser within one `game_id` (verified working via `guesses_aggregate`, which needed `allow_aggregations: true` on the select permission ‚Äî easy to miss) ‚Äî gives "resets every game" and "wiped on lobby close" for free via cascade deletes, while `posts` survive since they don't reference a lobby at all. Admin is derived (`profiles.is_admin`); Organizer is not (`lobbies.organizer_id`, mutable). On Hasura this maps to normal tables + row-level Hasura permissions instead of Postgres RLS policies. **Admin's own bypass path is built** (2026-07-08, #23): a `sync_admin_role` trigger on `profiles` (`AFTER INSERT OR UPDATE OF is_admin`) keeps `auth.user_roles` in sync automatically, granting/revoking a Hasura role whenever `is_admin` flips ‚Äî no app code involved. That role is named `app_admin`, not `admin` ‚Äî `admin` is a Hasura-reserved role name that always bypasses every permission check regardless of how it's granted, which would have handed out full unrestricted DB access (including `password_hash`) to anyone flagged as admin. `app_admin` has its own scoped permissions instead: unrestricted select on `lobbies`/`lobby_members`/`posts` (including `posts.submitted_by`, hidden from the `user` role) and `profiles`, plus select+delete on `auth.users` (delete has a self-delete guard via a row filter so Admin can't lock itself out).

**Frontend has no real-time push ‚Äî it polls.** The Nhost JS SDK doesn't support GraphQL subscriptions (only one-shot query/mutation), so the waiting room and live round both poll the backend every 2 seconds rather than getting pushed updates. At the ~50-person scale this project targets, that's plenty responsive and far simpler than hand-rolling a WebSocket client. One consequence worth knowing: the "current round" the client shows isn't simply "the first unrevealed round" ‚Äî it briefly holds on a just-revealed round (`REVEAL_HOLD_MS = 4000` in `app.js`) so players actually see the correct/wrong stamp before the view auto-advances; without this, a poll landing right after reveal would jump straight to the next round and the stamp would never be seen (a real bug found via live two-browser testing, not something reasoning-from-the-schema would have caught).

**Nhost's Auth service rate-limits per-IP, and it covers signIN too, not just signup** ‚Äî worth knowing before a heavy testing day: after enough signup/signin calls in a short window, `/v1/signup/email-password` *and* `/v1/signin/email-password` both start returning 429, and the app surfaces this as a generic "An unexpected error occurred." Two workarounds that don't touch the rate-limited Auth REST API at all, both used to finish verifying #32/#33 when this hit:
1. **Create test accounts by inserting directly into `auth.users`** (via the Hasura admin-secret SQL endpoint) with a `bcryptjs`-generated `password_hash` (`bcrypt.hashSync(password, 10)` ‚Äî Nhost's Go bcrypt implementation reads any standard bcrypt hash regardless of which library wrote it) instead of calling the signup endpoint. ## Real-usage fixes from actually playing the game on a real phone (2026-07-09)

Once the trainee had a real account promoted to `is_admin = true` (a plain `UPDATE profiles SET is_admin = true` for the specific row ‚Äî no dedicated flow, matches the "single admin account" design) and played a real game on a real device, two real problems surfaced that no amount of automated testing had caught:

- **The vote lineup was forced open the instant it became your turn to guess, permanently covering the video for the rest of the round.** The mockup's original design ‚Äî hidden by default, opened via a corner button, closed again by tapping outside or by locking in a guess ‚Äî had been dropped when the real game engine (#6) was built; the CSS for it (`.guess-toggle-btn`, `.guess-sheet.open`) was already sitting unused in `style.css` the whole time. Restored it in `app.js`/`index.html`. Fixing this also surfaced a real bug: the dimming backdrop (`.video-stage::before`) had `pointer-events: none`, so a tap meant to close the sheet fell through to the real TikTok iframe's own cross-origin content instead of registering on the page ‚Äî invisible in the mockup, which only ever had a decorative (non-iframe) video area, never a real cross-origin embed to fall through to.
- **Real TikTok links break in two separate ways the mockup never exercised, since it never touched a real video:**
  1. **Short share links** (`vm.tiktok.com/...`, `vt.tiktok.com/...` ‚Äî what TikTok's native "Share" button actually produces on a phone, as opposed to the long `tiktok.com/@user/video/123...` form) don't contain a numeric video ID anywhere in the URL, so the client-side ID-extraction regex always failed and the video silently fell back to a "Watch on TikTok" link instead of playing inline. Fixed with a new **Nhost Function**, `functions/resolve-tiktok-link.js` ‚Äî the first serverless function in this project ‚Äî which resolves the redirect **server-side** (a cross-origin redirect to tiktok.com is opaque to browser JS under CORS, so this genuinely can't be done client-side) and returns the canonical `https://www.tiktok.com/video/{id}` form. `app.js`'s submit flow calls it via `nhost.functions.post("/resolve-tiktok-link", {url})` before inserting, only for short-link hosts (long-form links skip the extra round-trip). Nhost Functions auto-deploy from a `functions/` folder in this same connected GitHub repo ‚Äî no separate deploy step, no Nhost CLI needed, consistent with how the schema/permissions are managed directly via the Hasura API rather than local migrations.
  2. **The embed itself showed way more than just the video.** `/embed/v2/{id}` (the endpoint originally chosen, see the game-mechanics section above) turned out to render TikTok's full social-post card ‚Äî caption, music-info line, like/comment/share counts, and a follow button ‚Äî not a minimal player, confirmed against a real video a real user actually played the game with. This was a genuine **game-design bug, not just visual clutter**: the caption showed the submitter's TikTok username directly on screen, which deanonymizes who submitted the video before the reveal, undermining the entire "hidden until reveal" mechanic the game is built around. Switched to `https://www.tiktok.com/player/v1/{id}?music_info=0&description=0` ‚Äî TikTok's dedicated **Embed Player**, a genuinely different, separate endpoint from the social-card embed, built specifically for this kind of third-party use. The two query params explicitly suppress the caption/music-info chrome it still shows by default.

**Environment limitation worth knowing:** this sandbox has no network access to any `tiktok.com` subdomain at all (confirmed directly ‚Äî general internet access works fine, but `curl` to `www.tiktok.com` *and* `developers.tiktok.com` both hang and fail to connect). This means TikTok's actual visual embed behavior can never be verified by an AI working in this same kind of sandboxed environment ‚Äî it can only be checked on a real device. Everything *up to* that boundary (the app's own DOM behavior, the resolve function's correctness, the exact URL constructed) was still verified directly ‚Äî via Playwright screenshots for the DOM/toggle behavior, and by calling the deployed Nhost Function directly (which runs on Nhost's own infrastructure, not this sandbox, so *it* can reach tiktok.com fine) to confirm the redirect resolution actually works against a real short link a real user had submitted.
