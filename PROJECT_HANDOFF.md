## Who's building this
An IT trainee, new to the job, with school-level coding experience and some basic Java — this is their first real personal project, built as a learning exercise with AI help. **No prior web/backend experience.** Explain *why*, not just *what*. Present options with tradeoffs rather than assuming.

## The idea
Inspired by a TikTok showing friends arguing over who liked a particular video in a shared "Liked Videos" tab — reimagined as a small multiplayer party game. Players secretly submit videos they've liked; the group guesses who submitted each one as it's drawn.

## How the game works
1. Players submit links to videos they've personally liked into their own private pool (no list of your own past submissions is ever shown back to you). Other players don't learn who submitted a video until it's drawn into a game and revealed — Admin can see the mapping anytime, for moderation.
2. Players join or create a **lobby** via a 6-digit code and land in a waiting room until the Organizer starts a game.
3. A game is a fixed **8 rounds**. Each round draws an unused video from the members' combined pools; everyone guesses who liked it; once everyone's guessed, the answer and updated standings reveal at once.
4. After round 8, final standings show. The same lobby can replay — standings reset to zero each new game.

## Why self-submission (not scraping TikTok)
TikTok has no public API for reading a user's Liked Videos or inbox/DMs. Scraping a logged-in session (directly, or via a shared "burner" account) would violate TikTok's ToS, risk a ban, and rely on fragile private endpoints — ruled out entirely. Instead, each player pastes a link to their own liked video directly into the app (a normal DB write, zero TikTok contact). Owner and submitter are always the same person.

*Possible future enhancement, not MVP:* a PWA Web Share Target so Android/Chrome users can tap Share on a TikTok video and land pre-filled in the submit form (iOS/Safari doesn't support this — manual paste stays as the universal fallback either way).

## Roles
- **Guest** — hasn't registered/logged in yet. Can only register or log in; every other tab is visible but locked until then.
- **Player** — submits videos, joins/creates lobbies, guesses each round, views standings, edits their own profile, can leave a lobby.
- **Organizer** — per-lobby (not global). Starts as the creator; if they leave, the role transfers to whoever joined that lobby earliest among the remaining members. Can start games, invite via the join code, and **kick a player — waiting-room only, not mid-game** (not a ban; they can rejoin with the code). Does not manage the video pool — that's Admin's job.
- **Admin** — global, just the trainee's own account (`is_admin` flag). Can view/close any lobby, manage the player roster (including deleting an account entirely — a destructive action, confirmed before it happens), and moderate any submitted video.

Diagram: [`roles-use-cases-diagram.md`](./roles-use-cases-diagram.md) / `.mmd`.

## Lobby lifecycle
Lobbies are temporary — they only exist while they have ≥1 member.
- Leaving/kicking works the same way: remove that person's membership; if they were Organizer, the role transfers per the rule above.
- **Last member leaves → the lobby soft-closes** (`is_open = false`, row kept). Its guesses/points/standings are wiped (always scoped to a live lobby, never persist beyond it) — but every submitted video (`posts`) survives regardless of which lobby held it, unless Admin deletes it directly.
- **No joining mid-game:** once Start Game is clicked, that lobby's code stops working for new members until the game finalizes.

## Tech stack
- **Frontend:** plain HTML/CSS/JS, no framework, no build step. `@nhost/nhost-js` loaded via ESM CDN (`esm.sh`) directly in `app.js`.
- **Backend/Auth/DB:** [Nhost](https://nhost.io) — free, open-source Firebase alternative built on Hasura GraphQL + Postgres. Bundles auth + a GraphQL API, both callable directly from frontend JS via `@nhost/nhost-js` — no custom server needed. **Switched from an earlier Supabase plan (2026-07-08)** — same rationale (free, open-source, no credit card), the trainee just ended up signing up with Nhost instead.
- **Hosting: GitHub Pages** — **decided and deployed 2026-07-08**, live at **https://chrylox.github.io/who-liked-it/**. Chosen over Netlify/Vercel (would need the trainee to personally sign up and hand over an API token — no account-creation/OAuth possible on an assistant's behalf) and over Scaleway, a genuinely EU-headquartered option that was specifically considered but requires a non-prepaid credit card just to create an account, with no free-forever tier — breaking the same "free, no card" principle that drove the Nhost choice. GitHub Pages needed the repo flipped from private to public (done with explicit sign-off) but needed no new signups. Nhost is backend-only (auth/DB/storage/functions) and does not host static sites — this was a deliberate, separate piece of infrastructure, the same pattern as e.g. Supabase+Vercel.
- **Multi-user:** real per-person accounts, not a shared device.

## Database schema (Nhost/Postgres via Hasura, 7 tables)
- **`profiles`** ✅ **built 2026-07-08** — `id` (PK, → `auth.users.id`, cascade delete), `is_admin` (bool, default false). No `display_name` here — Nhost's own `auth.users` already has `display_name`/`avatar_url`, so we didn't duplicate it; a Hasura select permission exposes just those two columns of `auth.users` to any logged-in player instead. A Postgres trigger (`on_auth_user_created`) auto-creates the matching `profiles` row on every signup, so app code never has to remember to.
- **`lobbies`** ✅ **built 2026-07-08** — `id`, `code` (unique 6-digit), `name`, `organizer_id` (→ profiles, mutable/current), `created_by` (→ profiles, immutable/history), `created_at`, `is_open`.
- **`lobby_members`** ✅ **built 2026-07-08** — `lobby_id`, `user_id`, `joined_at` (drives Organizer-transfer order and "last member left" detection).
- **`posts`** ✅ **built 2026-07-08** — a submitted video: `submitted_by` (→ profiles, `ON DELETE SET NULL` — if an account is later deleted, its past videos stay in history as an orphaned/"removed user" row rather than corrupting old games), video URL, timestamp. No `lobby_id` — it's a personal pool. Never deleted automatically — only Admin can remove one, via moderation (not built yet). **`submitted_by` is not exposed to the `user` role at all** — the whole secret-submitter mechanic depends on that column never being queryable directly (see reveal mechanism below). `UNIQUE (submitted_by, video_url)` prevents a person submitting the same link twice — deliberately scoped per-person, so two different friends independently liking the same video is still allowed (github.com/chrylox/who-liked-it/issues/24).
- **`games`** ✅ **built 2026-07-08** — one row per game session in a lobby: `id`, `lobby_id`, `started_at`, `finalized_at` (null while in progress), `round_count`. Revised from a fixed "always 8" — **`round_count` now shrinks to fit the pool** (`LEAST(8, distinct videos available across current lobby members)`), so a small/new group can still play instead of being blocked. A partial unique index guarantees only one *active* (unfinalized) game per lobby at a time.
- **`game_rounds`** ✅ **built 2026-07-08** — `id`, `game_id`, `round_number`, `post_id` (the video drawn), `revealed_at`.
- **`guesses`** ✅ **built 2026-07-08** — `round_id`, `guesser_id`, `guessed_user_id`, `created_at`, `correct` (bool, computed server-side by trigger — never client-set). Primary key is `(round_id, guesser_id)`, no surrogate id, matching the original schema note (one guess per person per round).

**How starting/playing a game actually works (all server-side, not client-driven inserts):**
- **Drawing rounds is a Postgres function (`start_game`), not a client INSERT.** A client that could freely INSERT `games`/`game_rounds` rows could rig the draw (e.g. an Organizer always drawing a video whose submitter they already know). `start_game(lobby_id)` is called as a Hasura *mutation*; it uses Hasura's `session_argument` feature to receive the caller's real user id server-side (can't be spoofed the way a plain argument could), checks they're that lobby's Organizer, computes `round_count = LEAST(8, pool size)`, and atomically inserts the game plus all rounds with `ORDER BY random()`.
- **A submitter never guesses on their own round** (confirmed with the trainee 2026-07-08) — enforced twice: the reveal-trigger's "expected guesser count" excludes them, and a guess `INSERT` from the actual submitter is rejected outright by a trigger, so the UI *and* the API both prevent it.
- **Reveal is fully automatic, not an explicit action.** A trigger on `guesses` (`AFTER INSERT`) counts actual guesses for that round against the expected count (lobby member count minus the submitter) and sets `revealed_at = now()` the moment they match — no client ever sets `revealed_at`. When the round that was just revealed is the game's last round, the same trigger sets `games.finalized_at`, auto-finalizing.
- **Guessing is sequential, enforced by trigger, not just UI:** a guess on round *N* is rejected if any round before it in the same game isn't revealed yet, and rejected outright if round *N* itself is already revealed.
- **Membership is locked for the duration of a game:** `lobby_members` insert (join) and delete (leave/kick) permissions both now additionally require *no* unfinalized game exists for that lobby — otherwise the "expected guesser count" the reveal trigger relies on could shift mid-game. Unlocks again automatically once the game finalizes (verified with a real replay in testing).
- **`posts.submitted_by` is only ever revealed through the `correct` guess field after `revealed_at` is set** (`guesses` select permission requires `round.revealed_at IS NOT NULL`) — nobody, including the guesser themselves, can peek at whether their own just-submitted guess was correct before everyone's guessed and the round officially reveals.

Standings = aggregate `guesses.correct` grouped by guesser within one `game_id` (verified working via `guesses_aggregate`, which needed `allow_aggregations: true` on the select permission — easy to miss) — gives "resets every game" and "wiped on lobby close" for free via cascade deletes, while `posts` survive since they don't reference a lobby at all. Admin is derived (`profiles.is_admin`); Organizer is not (`lobbies.organizer_id`, mutable). On Hasura this maps to normal tables + row-level Hasura permissions instead of Postgres RLS policies. **Admin's own bypass path (a Hasura role derived from `profiles.is_admin`, e.g. via Nhost Auth custom claims) is not built yet** — today only the raw Hasura admin secret can act as Admin; building the real Admin panel needs this first.
